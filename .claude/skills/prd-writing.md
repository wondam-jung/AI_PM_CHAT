# PRD (Product Requirements Document) 작성

## 목적
개발팀, 디자인팀, 이해관계자가 제품을 동일하게 이해하고 실행할 수 있도록
명확하고 구체적인 요구사항을 문서화한다.

## PRD 구성 섹션

### 1. 개요 (Overview)
- **문서 버전**: v1.0
- **작성일**: [날짜]
- **작성자**: [이름]
- **상태**: 초안 / 검토 중 / 확정
- **한 줄 요약**: [이 PRD가 다루는 내용을 한 문장으로]

### 2. 배경 및 목적 (Background & Purpose)
- **왜 이 기능이 필요한가?**: 비즈니스 맥락과 데이터
- **해결하려는 문제**: 고객 또는 비즈니스 문제 명시
- **목표**: 이 기능으로 달성하고자 하는 것

### 3. 목표 사용자 (Target Users)
- 주요 사용자 페르소나 (1-2개)
- 사용 시나리오: "누가, 언제, 어떤 상황에서 이 기능을 사용하는가"
- 사용자 스토리 (핵심 1-3개)

### 4. 기능 요구사항 (Functional Requirements)

**Must Have (필수)**
- 반드시 포함해야 하는 핵심 기능
- 없으면 제품이 동작하지 않는 기능

**Should Have (권장)**
- 있으면 크게 향상되지만 없어도 동작 가능한 기능
- 다음 버전에 포함 고려

**Nice to Have (선택)**
- 있으면 좋지만 우선순위 낮은 기능
- 리소스 여유 시 구현

### 5. 비기능 요구사항 (Non-Functional Requirements)
- **성능**: 페이지 로드 시간, API 응답 시간
- **보안**: 인증, 권한, 데이터 보호
- **가용성**: 업타임 목표
- **확장성**: 동시 사용자 수, 데이터 규모
- **접근성**: WCAG 기준

### 6. 범위 외 (Out of Scope)
- 이번 버전에서 의도적으로 제외한 기능
- 오해를 방지하기 위해 명시적으로 기록

### 7. 성공 지표 (Success Metrics)
- **주요 지표 (Primary Metric)**: 이 기능의 성공을 판단하는 핵심 수치
- **보조 지표 (Secondary Metrics)**: 2-3개 추가 지표
- **가드레일 지표 (Guardrail Metrics)**: 악화되면 안 되는 지표
- **측정 방법**: 어떻게 데이터를 수집할 것인가

### 8. 의존성 및 리스크 (Dependencies & Risks)
- **기술적 의존성**: 필요한 인프라, API, 서비스
- **팀 간 의존성**: 다른 팀의 협조가 필요한 사항
- **주요 리스크**: 실현 가능성, 일정, 기술 리스크
- **완화 방안**: 각 리스크 대응 전략

### 9. 타임라인 (Timeline)
- 주요 마일스톤 및 예상 완료일
- 리뷰 및 승인 일정

## 산출물 형식

```
# PRD: [기능명]

## 개요
- 버전: v1.0 | 작성일: [날짜] | 상태: 초안
- 한 줄 요약: [요약]

## 배경
[문제 상황 및 데이터]

## 목표 사용자
[페르소나 및 사용 시나리오]

## 기능 요구사항
### Must Have
- [ ] [기능 1]
- [ ] [기능 2]

### Should Have
- [ ] [기능 3]

## 비기능 요구사항
[성능, 보안, 접근성]

## 범위 외
- [제외 항목 1]

## 성공 지표
- 주요: [지표 및 목표값]
- 보조: [지표들]

## 리스크
[리스크 및 완화 방안]
```
