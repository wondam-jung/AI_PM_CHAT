# 유저 스토리 작성

## 목적
사용자 관점에서 기능을 정의하고, 개발팀과 PM이 동일한 이해를 공유하며 개발 가능한 수준의 구체적인 스토리를 작성한다.

## 유저 스토리 기본 형식
```
나는 [사용자 유형]으로서,
[목표/원하는 것]을 위해
[기능/액션]을 원한다.
```

## INVEST 원칙 체크리스트

| 원칙 | 설명 | 체크 |
|------|------|------|
| **I**ndependent | 다른 스토리와 독립적으로 개발 가능 | ☐ |
| **N**egotiable | 세부 구현 방식은 팀이 협의 가능 | ☐ |
| **V**aluable | 사용자 또는 비즈니스에 명확한 가치 제공 | ☐ |
| **E**stimable | 개발팀이 크기를 추정할 수 있음 | ☐ |
| **S**mall | 1-2 스프린트 내 완료 가능한 크기 | ☐ |
| **T**estable | 완료 여부를 테스트로 확인 가능 | ☐ |

## 인수 조건 (Acceptance Criteria) 작성

**Given-When-Then 형식:**
```
Given [사전 조건/컨텍스트]
When [사용자 액션]
Then [예상 결과/시스템 반응]
```

**예시:**
```
Given 로그인된 사용자가 대시보드에 있을 때
When "새 OKR 추가" 버튼을 클릭하면
Then OKR 입력 폼이 나타나고 Objective 필드에 포커스가 이동한다
```

## 스토리 크기 (Story Points) 기준

| 포인트 | 설명 | 예시 |
|--------|------|------|
| 1 | 매우 간단한 변경 | 텍스트 수정, 색상 변경 |
| 2 | 간단한 기능 | 버튼 추가, 폼 필드 추가 |
| 3 | 보통 복잡도 | 새로운 화면, 간단한 API 연동 |
| 5 | 높은 복잡도 | 복잡한 비즈니스 로직, 여러 API |
| 8 | 매우 복잡 | 신규 서비스 통합, 아키텍처 변경 |
| 13 | 에픽 분리 필요 | 너무 크면 반드시 분리 |

## 단계별 수행 방법

### 1단계: 에픽(Epic) 정의
큰 기능 단위를 먼저 에픽으로 정의한다.
형식: "에픽: [에픽명] - [한 줄 설명]"

### 2단계: 스토리 분해
에픽을 독립적인 유저 스토리로 분해한다.
분해 기준:
- 사용자 액션 단위
- 데이터 처리 단계 단위
- CRUD(생성/조회/수정/삭제) 단위
- 역할(Role) 단위

### 3단계: INVEST 검증
각 스토리에 대해 INVEST 체크리스트 검토

### 4단계: 인수 조건 작성
각 스토리별로 Given-When-Then 형식으로 2-5개 AC 작성

### 5단계: 정의 완료 (Definition of Done)
모든 스토리에 공통으로 적용되는 완료 기준:
- [ ] 코드 리뷰 완료
- [ ] 단위 테스트 통과
- [ ] QA 테스트 통과
- [ ] 기획 검수 완료
- [ ] 스테이징 배포 확인

## 산출물 형식

```
## 에픽: [에픽명]

### 유저 스토리 1: [스토리 제목]
**스토리:**
나는 [사용자 유형]으로서,
[목표]를 위해
[기능]을 원한다.

**스토리 포인트:** [1/2/3/5/8]
**우선순위:** Must Have / Should Have

**인수 조건:**
1. Given [컨텍스트] When [액션] Then [결과]
2. Given [컨텍스트] When [액션] Then [결과]
3. (엣지 케이스) Given [에러 상황] When [액션] Then [에러 처리]

**INVEST 체크:**
- [I] 독립적: ✅
- [N] 협상 가능: ✅
- [V] 가치 있음: ✅
- [E] 추정 가능: ✅
- [S] 적정 크기: ✅
- [T] 테스트 가능: ✅
```
